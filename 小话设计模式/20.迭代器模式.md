## 1. 通俗易懂的例子：回转寿司店 🍣

想象你在一家回转寿司店用餐。

* **❌ 错误的做法（不使用模式）：**
    你想吃寿司，但老板让你自己进厨房翻冰箱。你需要知道：三文鱼放在冷冻柜第三层，鳗鱼在料理台下方的盒子里，米饭在电饭煲里。
    * **麻烦：** 你只是个食客，却被逼着了解厨房的每一个角落。如果老板明天换了装修，把三文鱼改放到地窖，你就找不到吃的了。这叫“暴露了内部构造”，对你和老板都很累。

* **✅ 正确的做法（迭代器模式）：**
    老板在你面前放了一条**传送带**。
    * **巧妙之处：** 所有的寿司依次经过你面前。你不需要知道寿司是怎么做的，也不需要知道它们原本藏在厨房哪里。你只需要做一个动作：**“拿下一盘，吃掉；再等下一盘”**。
    * **结果：** 厨房内部再怎么乱（内部数据结构是数组、链表还是树），对你来说都只是一条简单的传送带（统一的遍历接口）。

## 2. Unity 游戏实战场景：回合制游戏的单位行动管理 ⚔️

在回合制游戏（如《神界：原罪》或《崩坏：星穹铁道》）中，**管理行动顺序（Turn Order）** 是迭代器模式的最佳舞台。

* **为什么要用？**
    战场上的单位可能存在于不同的地方：有些在“玩家队列”里，有些在“敌人数组”里，有些可能是中立召唤物。
* **痛点：**
    作为回合管理器（TurnManager），我不关心你们这些单位是存在 List 里还是 Dictionary 里。我只想要一个“传送带”，能让我一个接一个地通知单位：“轮到你了，请出招”。
* **应用效果：**
    迭代器模式把复杂的战斗单位存储逻辑隐藏起来，给管理器提供一个干净的“下一个是谁”的接口。

## 3. C# 代码实现（分步讲解）

虽然 C# 自带 `IEnumerator`，但为了让你理解原理，我们手动模拟一个简单的版本。

### 第一步：定义迭代器接口（传送带的标准）

```csharp
// 📜 迭代器接口：规定了怎么“拿下一个”
public interface IMyIterator
{
    bool HasNext();   // 还有下一盘寿司吗？
    object Next();    // 把下一盘寿司递给我
}

// 📦 容器接口：规定了怎么“生产传送带”
public interface ICollection
{
    IMyIterator CreateIterator();
}
```

### 第二步：编写具体的实现类（战斗单位和战场）

```csharp
using UnityEngine;
using System.Collections.Generic;

// 🛡️ 战斗单位
public class BattleUnit
{
    public string Name;
    public BattleUnit(string name) => Name = name;
}

// 🏟️ 战场类：这就是我们的“厨房”，内部存了很多单位
public class BattleField : ICollection
{
    private List<BattleUnit> units = new List<BattleUnit>();

    public void AddUnit(BattleUnit unit) => units.Add(unit);

    // 核心：战场产生一个专门遍历自己的迭代器
    public IMyIterator CreateIterator()
    {
        return new BattleFieldIterator(units);
    }

    // 第三步：具体的迭代器实现（传送带的机械逻辑）
    private class BattleFieldIterator : IMyIterator
    {
        private List<BattleUnit> _units;
        private int _index = 0; // 当前指到哪了

        public BattleFieldIterator(List<BattleUnit> units) => _units = units;

        public bool HasNext() => _index < _units.Count;

        public object Next()
        {
            if (HasNext())
            {
                return _units[_index++]; // 返回当前单位并把指针后移
            }
            return null;
        }
    }
}
```

## 4. 在 Unity 中如何使用？

我们通过 `TurnManager` 脚本来模拟遍历战场单位的过程。

```csharp
using UnityEngine;

public class TurnManager : MonoBehaviour
{
    void Start()
    {
        // 1. 准备厨房（战场）并添加单位
        BattleField battleField = new BattleField();
        battleField.AddUnit(new BattleUnit("亚瑟"));
        battleField.AddUnit(new BattleUnit("梅林"));
        battleField.AddUnit(new BattleUnit("恶龙"));

        // 2. 获取传送带（迭代器）
        IMyIterator iterator = battleField.CreateIterator();

        Debug.Log("--- 回合开始：按顺序行动 ---");

        // 3. 只要传送带上还有东西，就一直拿
        while (iterator.HasNext())
        {
            BattleUnit unit = (BattleUnit)iterator.Next();
            Debug.Log($"[通知] 📢 轮到单位：{unit.Name} 行动了！");
        }
        
        Debug.Log("--- 所有单位行动结束 ---");
    }
}

/* 预期 Debug.Log 输出结果：
1. --- 回合开始：按顺序行动 ---
2. [通知] 📢 轮到单位：亚瑟 行动了！
3. [通知] 📢 轮到单位：梅林 行动了！
4. [通知] 📢 轮到单位：恶龙 行动了！
5. --- 所有单位行动结束 ---
*/
```

## 5. 核心原理与优缺点总结

### 💡 它解决了什么痛点？
它解决了 **“遍历方式与数据结构强耦合”** 的问题。有了迭代器，无论你的数据是存在数组、链表、还是复杂的二叉树里，外部调用者（比如 `foreach`）都只需要用同一种方式来访问，而不需要了解内部的复杂逻辑。

### ✅ 优点
1.  **封装性好：** 隐藏了容器内部的结构，保护了数据。
2.  **统一接口：** 为不同的集合提供了通用的遍历方式。
3.  **多重遍历：** 一个容器可以同时产生多个迭代器，它们之间互不干扰（比如两个玩家同时翻阅同一个背包）。

### ❌ 缺点
1.  **类变多了：** 每个容器都要对应一个迭代器类。
2.  **效率微降：** 对于极简单的数组遍历，用迭代器会比直接 `for` 循环多一点点开销（在 Unity 大多数场景下可忽略不计）。

### 🎮 Unity 中的其他常见场景
* **物品栏（Inventory）：** 遍历背包里所有的道具。
* **对话系统（Dialogue）：** 一条接一条地显示文本。
* **协程（Coroutine）：** Unity 的 `yield return` 本质上就是利用 C# 的 `IEnumerator` 迭代器实现的。

### ⚠️ 导师点拨：它和“访问者模式”的区别
一句话点明核心区别：
> **迭代器模式** 目的是 **“一个接一个地拿到对象”**；而 **访问者模式** 目的是 **“给对象增加新操作”**（比如对每一个拿到的单位进行伤害计算）。

在 C# 中，你通常不需要手动写迭代器类，直接使用 `IEnumerable` 和 `yield return` 就可以实现同样的效果，那才是 Unity 开发者的“完全体”姿态！加油！🚀
