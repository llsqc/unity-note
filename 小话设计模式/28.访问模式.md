你好！我是你的 Unity 开发导师。今天我们要挑战的是 23 种设计模式中最“绕”但也最能体现系统美感的：**访问者模式（Visitor Pattern）**。

如果你发现自己经常要在很多不同的类（比如战士、法师、怪物）里反复添加相似的逻辑（比如“导出数据”、“计算伤害”），导致这些类变得越来越臃肿，那么访问者模式就是你的救星。

---

## 1. 通俗易懂的例子：医院体检 🏥

想象你去医院做入职体检。体检中心有很多科室：抽血处、心电图室、视力检查室。

* **❌ 错误的做法（不使用模式）：**
    医院要求每个病人自己学会“抽血”、“做心电图”。于是你（病人对象）的代码里塞满了：`DoBloodTest()`、`DoECG()`。
    * **麻烦：** 如果医院新增加了一项“核酸检测”，所有的病人（类）都得重新手术，给自己加一个 `DoPCR()` 的功能。病人本来只需要“活着”就行，现在却要学一堆医疗技能，逻辑太乱了！

* **✅ 正确的做法（访问者模式）：**
    你只需要静静地坐在那里（被访问者），医院派**医生（访问者）** 来找你。
    * **巧妙之处：** 抽血医生来了，他知道怎么抽你的血；眼科医生来了，他知道怎么查你的视力。你只需要提供一个统一的接口：“医生，我在这，你**访问**我吧（Accept）”。
    * **结果：** 增加新体检项目？只需要新派一种医生就行。你（病人类）的代码一行都不用改，非常干净。

## 2. Unity 游戏实战场景：多种类怪物的成就统计系统 🏆

在 Unity 游戏中，你可能有很多种怪物类：`Slime`（史莱姆）、`Orc`（兽人）、`Dragon`（巨龙）。

* **场景描述：**
    现在策划提出了两个需求：
    1.  统计当前场景所有怪物的总血量。
    2.  根据怪物的类型，分别触发不同的“击杀成就”。
* **为什么适合？**
    怪物的种类（数据结构）通常是比较稳定的，但你对怪物的“操作逻辑”却经常变。如果把统计、成就、数据分析逻辑全写在怪物类里，怪物类会变得像垃圾堆。
    使用访问者模式，你可以把这些“外来逻辑”抽离出来，让不同的“访问者”去处理。

## 3. C# 代码实现（分步讲解）

### 第一步：定义访问者接口与元素基类

```csharp
// 👨‍⚕️ 访问者接口：定义了访问不同怪物时的行为
public interface IMonsterVisitor
{
    void Visit(Slime slime);
    void Visit(Orc orc);
}

// 👾 怪物基类
public abstract class Monster
{
    public string Name;
    public int Health;

    // 🚪 核心：接受访问的入口
    public abstract void Accept(IMonsterVisitor visitor);
}
```

### 第二步：编写具体的怪物实现类

```csharp
using UnityEngine;

public class Slime : Monster
{
    public override void Accept(IMonsterVisitor visitor)
    {
        // 🤝 史莱姆告诉访问者：我是史莱姆，请处理我
        visitor.Visit(this);
    }
}

public class Orc : Monster
{
    public override void Accept(IMonsterVisitor visitor)
    {
        // 🤝 兽人告诉访问者：我是兽人，请处理我
        visitor.Visit(this);
    }
}
```

### 第三步：编写具体的访问者（成就计算器）

```csharp
using UnityEngine;

// 🏆 成就访问者：专门负责根据怪物类型处理成就
public class AchievementVisitor : IMonsterVisitor
{
    public void Visit(Slime slime)
    {
        Debug.Log($"[成就] 捏扁了史莱姆 {slime.Name}！获得成就：初级猎人 🎖️");
    }

    public void Visit(Orc orc)
    {
        Debug.Log($"[成就] 击败了兽人 {orc.Name}！获得成就：勇者之路 ⚔️");
    }
}

// 📊 统计访问者：专门负责算血量（展示逻辑分离）
public class HPStatsVisitor : IMonsterVisitor
{
    public int TotalHP = 0;

    public void Visit(Slime slime) => TotalHP += slime.Health;
    public void Visit(Orc orc) => TotalHP += orc.Health;
}
```

## 4. 在 Unity 中如何使用？

```csharp
using UnityEngine;
using System.Collections.Generic;

public class VisitorTest : MonoBehaviour
{
    void Start()
    {
        // 1. 准备一堆怪物（病人）
        List<Monster> monsterList = new List<Monster>
        {
            new Slime { Name = "小绿", Health = 10 },
            new Orc { Name = "格罗姆", Health = 100 }
        };

        // 2. 创建一个成就访问者（医生A）
        AchievementVisitor achVisitor = new AchievementVisitor();

        Debug.Log("--- 开始发放成就 ---");
        foreach (var m in monsterList)
        {
            // 3. 怪物接受访问者的访问
            m.Accept(achVisitor);
        }

        // 4. 创建一个统计访问者（医生B）
        HPStatsVisitor hpVisitor = new HPStatsVisitor();
        foreach (var m in monsterList)
        {
            m.Accept(hpVisitor);
        }
        Debug.Log($"--- 场景怪物总血量: {hpVisitor.TotalHP} ---");
    }
}

/* 预期 Debug.Log 输出结果：
1. --- 开始发放成就 ---
2. [成就] 捏扁了史莱姆 小绿！获得成就：初级猎人 🎖️
3. [成就] 击败了兽人 格罗姆！获得成就：勇者之路 ⚔️
4. --- 场景怪物总血量: 110 ---
*/
```

## 5. 核心原理与优缺点总结

### 💡 它解决了什么痛点？
解决了 **“数据结构”与“业务算法”紧密耦合** 的问题。它让你可以在不修改已有类（如怪物类）的前提下，定义作用于这些类的新操作。

### ✅ 优点
1.  **极好的扩展性：** 增加新的操作（访问者）非常简单。
2.  **单一职责：** 怪物类只负责怪物的数据，成就逻辑归成就类，统计逻辑归统计类。
3.  **双分派巧妙：** 通过 `Accept` 方法，程序能在运行时自动找到对应的处理逻辑。

### ❌ 缺点
1.  **增加新数据结构很难：** 如果你新加了一个“巨龙”怪物，你得去修改所有的 `IMonsterVisitor` 接口。
2.  **破坏封装：** 访问者为了干活，通常需要怪物类暴露一些原本应该是 `private` 的内部数据。

### 🎮 Unity 中的其他常见场景
* **存档系统：** 不同类型的物品（武器、消耗品）接受一个“序列化访问者”来保存数据。
* **关卡导出工具：** 在编辑器模式下，不同的 GameObject 接受一个“XML 导出访问者”。
* **技能效果结算：** 不同的角色（玩家、NPC、陷阱）接受一个“范围伤害访问者”。

### ⚠️ 导师点拨：它和“策略模式”的区别
一句话点明核心区别：
> **策略模式**是让对象**选择**一种算法来执行（针对同一种行为的不同实现）；而**访问者模式**是让算法**作用于**一组复杂的不同类对象（针对不同种类的对象执行不同的逻辑）。

掌握了访问者模式，你就学会了如何优雅地管理大型项目中的复杂逻辑分发。加油！🚀
