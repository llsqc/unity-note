你好！我是你的 Unity 开发导师。今天我们要学习的是设计模式中的“社交达人”——**观察者模式（Observer Pattern）**。

在 Unity 开发中，如果你希望一个事件发生时（比如玩家死掉），全城都知道这个消息（UI 变灰、音效播放、成就解锁），但又不想让“玩家”脚本认识这么多乱七八糟的系统，那么观察者模式就是你的不二之选。

---

## 1. 通俗易懂的例子：订阅 B 站 UP 主 📺

想象你在 B 站（或者 YouTube）上关注了你最喜欢的游戏区 UP 主。

* **❌ 错误的做法（轮询模式）：**
    你每天每隔 10 分钟就打开 UP 主的主页刷一下，看看他更新了没有。
    * **麻烦：** 你太累了，大部分时间都是白跑一趟。如果有 100 个粉丝都这么干，UP 主的服务器（电脑）也会被你们问候得直接宕机。在代码里，这就是在 `Update` 里疯狂检查某个变量。

* **✅ 正确的做法（观察者模式）：**
    你点击了那个 **“小铃铛（订阅按钮）”** 。
    * **巧妙之处：** 你不需要再去盯着 UP 主了。UP 主发视频的一瞬间，平台会自动给所有点了铃铛的人发一条 **“推送通知”**。
    * **结果：** UP 主根本不需要知道每一个粉丝是谁（解耦），他只需要对着平台喊一声“我更新啦”，订阅者们自然会收到消息并做出反应（比如点进去看视频）。

## 2. Unity 游戏实战场景：成就与 UI 响应系统 🏆

在 Unity 中，**玩家属性变更（如死亡、升级、得分）** 是观察者模式最典型的舞台。

* **为什么要用？**
    假设玩家打碎了一个宝箱。
    1.  **UI 系统**要更新金币数字。
    2.  **成就系统**要检查“碎箱达人”进度。
    3.  **音效系统**要播放“叮”的一声。
* **如果不使用模式：**
    你的 `Chest` 脚本里得写：`uiManager.UpdateCoin()`、`achievementManager.Check()`、`audioSource.Play()`。
    **天呐！** 宝箱为什么要认识这么多经理？万一以后删掉成就系统，宝箱脚本就报错了。
* **使用模式：**
    宝箱只管发一个“我开了”的消息。谁感兴趣（订阅了），谁就自己去处理。

## 3. C# 代码实现（分步讲解）

为了让大家在 Unity 里直接能用，我们采用 C# 最推荐的 `event`（事件）方式来实现。

### 第一步：定义“发消息”的主体（UP 主）

```csharp
using UnityEngine;
using System; // 💡 使用 System.Action 需要这个命名空间

public class PlayerStats : MonoBehaviour
{
    // 🔔 定义一个事件：当血量改变时触发
    // Action<float> 表示这个事件会带一个 float 参数（当前的血量百分比）
    public static event Action<float> OnHealthChanged;

    private float _maxHealth = 100f;
    private float _currentHealth = 100f;

    // 🩸 模拟受伤的方法
    public void TakeDamage(float damage)
    {
        _currentHealth -= damage;
        _currentHealth = Mathf.Clamp(_currentHealth, 0, _maxHealth);

        Debug.Log($"[玩家] 哎哟！受了 {damage} 点伤。");

        // 📢 重点：发通知！
        // 如果有人订阅了（不为 null），就告诉他们最新的血量百分比
        OnHealthChanged?.Invoke(_currentHealth / _maxHealth);
    }
}
```

### 第二步：编写“收消息”的观察者（粉丝 A：成就系统）

```csharp
public class AchievementSystem : MonoBehaviour
{
    private void OnEnable()
    {
        // 📥 订阅：把自己的方法“挂”在事件上
        PlayerStats.OnHealthChanged += CheckDeathAchievement;
    }

    private void OnDisable()
    {
        // 📤 取消订阅：一定要记得在销毁或禁用时拔掉，否则会内存泄漏！
        PlayerStats.OnHealthChanged -= CheckDeathAchievement;
    }

    private void CheckDeathAchievement(float healthPercent)
    {
        if (healthPercent <= 0)
        {
            Debug.Log("[成就] 达成成就：『第一次白给』！🏅");
        }
    }
}
```

### 第三步：编写另一个观察者（粉丝 B：健康条 UI）

```csharp
public class HealthBarUI : MonoBehaviour
{
    private void OnEnable()
    {
        PlayerStats.OnHealthChanged += UpdateBloodBar;
    }

    private void OnDisable()
    {
        PlayerStats.OnHealthChanged -= UpdateBloodBar;
    }

    private void UpdateBloodBar(float percent)
    {
        Debug.Log($"[UI] 血条平滑缩放到: {percent * 100}% 💉");
    }
}
```

## 4. 在 Unity 中如何使用？

我们将这些脚本挂在场景中的不同物体上，运行测试。

```csharp
public class GameTester : MonoBehaviour
{
    private PlayerStats _player;

    void Start()
    {
        // 1. 获取玩家组件
        _player = FindObjectOfType<PlayerStats>();

        Debug.Log("--- 游戏开始，测试观察者模式 ---");

        // 2. 模拟受伤
        _player.TakeDamage(30f);
        
        Debug.Log("--- 玩家再次重伤 ---");
        _player.TakeDamage(70f);
    }
}

/* 预期 Debug.Log 输出结果：
1. --- 游戏开始，测试观察者模式 ---
2. [玩家] 哎哟！受了 30 点伤。
3. [UI] 血条平滑缩放到: 70% 💉
4. --- 玩家再次重伤 ---
5. [玩家] 哎哟！受了 70 点伤。
6. [UI] 血条平滑缩放到: 0% 💉
7. [成就] 达成成就：『第一次白给』！🏅
*/
```

## 5. 核心原理与优缺点总结

### 💡 它解决了什么痛点？
解决了 **“谁都要认识谁”** 的网状耦合。它让“被观察者”处于一种**高冷**的状态：它只管做自己的事，发自己的通知，完全不关心谁在听。

### ✅ 优点
1.  **极度解耦：** 增加一个新的 UI 或者是新的系统，完全不需要修改 `PlayerStats` 脚本。
2.  **一对多：** 一个事件可以触发无数个响应方法。
3.  **符合 OCP：** 对扩展开放（随便加新订阅者），对修改封闭（不改被观察者代码）。

### ❌ 缺点
1.  **内存泄露风险：** 如果对象被 Destroy 了但没取消订阅（`-=`），那个事件会一直抓着死掉的对象不放，导致报错或内存无法释放。
2.  **执行顺序不明：** 你没法简单地控制是先更新 UI 还是先检查成就。

### 🎮 Unity 中的其他常见场景
* **关卡管理器：** 当关卡倒计时结束，通知所有敌人消失、所有 UI 锁定。
* **设置界面：** 当玩家在菜单改了“画质”，所有场景中的光源自动调整亮度。
* **任务系统：** 怪物死亡发通知，任务脚本收到通知后更新“击杀数量”。

### ⚠️ 导师点拨：它和“中介者模式”的区别
一句话点明核心区别：
> **观察者模式**是“发广播”，消息是**单向**散播出去的；而**中介者模式**是“找接线员”，所有模块都把消息传给一个**中心调度员**，由它来协调多方复杂的对话。

掌握了观察者模式，你的代码逻辑就会变得像空气一样自由，再也不会牵一发而动全身了！加油！🚀
