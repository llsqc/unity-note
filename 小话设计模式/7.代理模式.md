你好！我是你的 Unity 开发导师。很高兴看到你已经掌握了装饰模式，现在我们要学习的**代理模式（Proxy Pattern）**，在结构上和装饰模式非常像，但在“心法”上却完全不同。

如果说装饰模式是给对象“穿衣服（加功能）”，那么代理模式就是给对象“找个经纪人（控制访问）”。

---

## 1. 通俗易懂的例子：大明星与经纪人 🕶️

想象你是一位世界级的游戏开发巨星。

* **❌ 错误的做法（不使用代理）：**
    你的手机号是公开的。无论是价值千万的合同谈判，还是路人甲问你“早饭吃了没”，都直接打到你手机上。
    * **麻烦：** 你每天忙于接听各种琐碎、骚扰电话，根本没时间静下心来写代码（核心业务被干扰）。

* **✅ 正确的做法（代理模式）：**
    你聘请了一位**经纪人（代理）**。对外只公布经纪人的电话。
    * **巧妙之处：** 1. **过滤筛选：** 经纪人接到电话，发现是骚扰电话直接挂掉。
        2. **权限控制：** 只有大项目合同，经纪人才会转交给你处理。
        3. **延迟满足：** 经纪人可以先替你答应一些琐事，等你有空了再统一处理。
    * **结果：** 外界觉得经纪人就代表你（接口一致），但你作为“本体”，被完美保护起来了。

## 2. Unity 游戏实战场景：大容量图片延迟加载 🖼️

在 Unity 中，代理模式最典型的应用之一就是**虚拟代理（Virtual Proxy）**，常用于 **UI 列表中的高清图标加载**。

* **为什么要用？**
    假设你有一个背包界面，里面有 100 把传说级武器，每个图标都是 4K 高清大图。
* **如果不使用代理：**
    打开背包的一瞬间，Unity 尝试同时加载 100 张大图，内存瞬间爆满，游戏直接卡死 2 秒。
* **使用代理：**
    我们不直接给 UI 传真正的图片，而是传一个“图片代理”。这个代理先显示一张模糊的占位图（加载快），只有当玩家真正滑动到这张图片时，代理才去后台加载真正的 4K 大图。

## 3. C# 代码实现（分步讲解）



### 第一步：定义共同的接口（“明星”和“经纪人”都得会演戏）

```csharp
// 🖼️ 无论是真图还是代理图，都得能被“显示”
public interface IImage
{
    void Display();
}
```

### 第二步：编写具体的实现类（“大明星”本体）

```csharp
using UnityEngine;

// 💾 真正的图片对象：加载它非常耗时且耗内存
public class RealHighResImage : IImage
{
    private string _imagePath;

    public RealHighResImage(string path)
    {
        _imagePath = path;
        LoadFromDisk(); // 构造时就开始沉重的加载
    }

    private void LoadFromDisk()
    {
        // 模拟耗时的 IO 操作
        Debug.Log($"[本体] 正在从硬盘读取超大文件: {_imagePath}... ⏳");
    }

    public void Display()
    {
        Debug.Log($"[本体] 完美呈现 4K 高清画质: {_imagePath} ✨");
    }
}
```

### 第三步：编写代理类（“经纪人”）

```csharp
// 代理类：平时它就是个“空壳”，只有关键时刻才召唤本体
public class ProxyImage : IImage
{
    private string _imagePath;
    private RealHighResImage _realImage; // 持有本体的引用

    public ProxyImage(string path)
    {
        _imagePath = path;
        // 💡 重点：构造时并不加载本体！
    }

    public void Display()
    {
        // 只有当真正需要显示时，才去创建和加载本体（延迟加载）
        if (_realImage == null)
        {
            _realImage = new RealHighResImage(_imagePath);
        }
        
        Debug.Log("[代理] 正在核对权限，准备调用本体显示...");
        _realImage.Display();
    }
}
```

## 4. 在 Unity 中如何使用？

我们写一个脚本模拟 UI 面板打开时的逻辑。

```csharp
using UnityEngine;

public class UILoader : MonoBehaviour
{
    void Start()
    {
        Debug.Log("--- 玩家打开了装备面板 ---");

        // 1. 创建代理对象
        // 此时并不会触发 RealHighResImage 的 LoadFromDisk，所以完全不卡顿
        IImage weaponIcon = new ProxyImage("HighRes_Sword_Icon.png");
        
        Debug.Log("--- 面板已打开，还没开始看具体武器 ---");

        // 2. 模拟玩家点击了某个图标，或者滑动到了该位置
        Debug.Log("--- 玩家点击了图标 ---");
        weaponIcon.Display(); // 此时才会真正触发加载

        // 3. 再次点击，因为本体已存在，不会重复加载
        weaponIcon.Display();
    }
}

/* 预期 Debug.Log 输出结果：
1. --- 玩家打开了装备面板 ---
2. --- 面板已打开，还没开始看具体武器 ---
3. --- 玩家点击了图标 ---
4. [本体] 正在从硬盘读取超大文件: HighRes_Sword_Icon.png... ⏳
5. [代理] 正在核对权限，准备调用本体显示...
6. [本体] 完美呈现 4K 高清画质: HighRes_Sword_Icon.png ✨
7. [代理] 正在核对权限，准备调用本体显示...
8. [本体] 完美呈现 4K 高清画质: HighRes_Sword_Icon.png ✨
*/
```

## 5. 核心原理与优缺点总结

### 💡 它解决了什么痛点？
解决了 **“直接访问对象带来的副作用”** 。比如对象太大加载慢（虚拟代理）、对象在远程服务器上访问不便（远程代理）、或者访问对象需要复杂的权限检查（保护代理）。

### ✅ 优点
1.  **性能优化：** 实现“按需加载（懒加载）”，节省内存和 CPU。
2.  **职责分离：** 本体只负责核心逻辑，代理负责权限、缓存、日志等杂事。
3.  **远程访问：** 可以让本地代码像访问本地对象一样访问网络对象。

### ❌ 缺点
1.  **响应延迟：** 如果代理需要实时创建本体，第一次访问可能会有轻微卡顿。
2.  **代码复杂：** 增加了一层结构，对于非常简单的对象来说没必要。

### 🎮 Unity 中的其他常见场景
* **网络代理：** 在多人游戏中，你看到的“队友”其实是网络代理，它接收服务器坐标并同步给本体模型。
* **权限保护：** 存档系统代理，只有当玩家通过验证后，代理才允许修改真正的存档文件。
* **日志代理：** 给某个核心系统（如战斗计算）加个代理，自动记录每次调用的输入输出，而不去污染核心算法。

### ⚠️ 导师点拨：它和“装饰模式”的区别
一句话点明核心区别：
> **装饰模式**是为了**增强功能**（你觉得它不够好，给它加点东西）；**代理模式**是为了**控制访问**（你觉得直接用它不方便/不安全，给它找个挡箭牌）。
