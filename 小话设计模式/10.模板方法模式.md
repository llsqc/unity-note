你好！我是你的 Unity 开发导师。今天我们要探讨的是一个非常有“秩序感”的设计模式——**模板方法模式（Template Method Pattern）**。

在 Unity 开发中，如果你发现自己写了好几个脚本，它们的执行逻辑大体一致，只有某些细节步骤不同，那么模板方法模式就是为你准备的“逻辑模具”。

---

## 1. 通俗易懂的例子：冲泡饮料的套路 ☕

想象你在经营一家自动饮品机，提供“冲咖啡”和“泡茶”两种服务。

* **❌ 错误的做法（不使用模式）：**
    你为咖啡机写了一套逻辑：烧水 -> 磨豆子 -> 倒进杯子 -> 加糖。
    又为泡茶机写了一套完全独立的逻辑：烧水 -> 浸泡茶叶 -> 倒进杯子 -> 加柠檬。
    * **麻烦：** 你发现“烧水”和“倒进杯子”这两步在两台机器里是一模一样的。如果你想把烧水温度从 **90°C** 改成 **100°C**，你得在两个脚本里都改一遍。逻辑越复杂，出错概率越大。

* **✅ 正确的做法（模板方法模式）：**
    你制定了一个 **“热饮冲泡流程模板”** ：
    1. 烧水（固定的）
    2. **冲泡（交给具体饮品决定）**
    3. 倒进杯子（固定的）
    4. **加调料（交给具体饮品决定）**
    * **巧妙之处：** 流程的大框架是死的，但中间留出了几个“填空题”。冲咖啡时填入“磨豆”，泡茶时填入“浸泡”。
    * **结果：** 核心逻辑由总部控制，细节由分部实现。

## 2. Unity 游戏实战场景：敌人 AI 行为流程 🤖

在 Unity 战斗系统中，**敌人 AI 的行为决策**是该模式最典型的应用。

* **为什么要用？**
    所有的敌人都遵循类似的逻辑循环：**观察环境 -> 移动 -> 攻击 -> 休息**。
    * “小怪”可能是：发现玩家 -> 走过去 -> 挥拳 -> 呆立 1 秒。
    * “远程兵”可能是：发现玩家 -> 保持距离 -> 射箭 -> 跑开。
* **目标：**
    我们在基类里写死“观察、移动、攻击”的**先后顺序**，而具体的敌人只需要去具体实现“怎么移动”和“怎么攻击”即可。这保证了所有 AI 的行为逻辑都是统一且可控的。

## 3. C# 代码实现（分步讲解）



### 第一步：定义抽象基类（制定“逻辑模具”）

```csharp
using UnityEngine;

// 🛡️ 敌人 AI 的模板基类
public abstract class EnemyAI
{
    // 💡 核心：这就是“模板方法”。
    // 用 sealed 关键字可以防止子类乱改这个流程顺序。
    public void ExecuteTurn()
    {
        LookAround();    // 第一步：观察
        Move();          // 第二步：移动
        Attack();        // 第三步：攻击
        Rest();          // 第四步：休息
        Debug.Log("--- 回合结束 ---");
    }

    // 固定的步骤，子类直接复用即可
    private void LookAround() => Debug.Log("正在扫描附近的玩家... 👀");
    private void Rest() => Debug.Log("进入休眠恢复体力... 💤");

    // 变化的步骤，定义为抽象方法，让子类去“填空”
    protected abstract void Move();
    protected abstract void Attack();
}
```

### 第二步：编写具体的实现类（实现“细节步骤”）

```csharp
// 🟢 近战小怪：傻乎乎地冲过去打
public class MeleeEnemy : EnemyAI
{
    protected override void Move() => Debug.Log("小怪：嘿咻嘿咻跑向玩家！🏃");
    protected override void Attack() => Debug.Log("小怪：吃我一记重拳！👊");
}

// 🏹 远程射手：猥琐发育
public class RangedEnemy : EnemyAI
{
    protected override void Move() => Debug.Log("射手：保持距离，向后拉扯。🏹");
    protected override void Attack() => Debug.Log("射手：百步穿杨，放箭！🎯");
}
```

## 4. 在 Unity 中如何使用？

我们写一个简单的管理器来驱动这些 AI 的逻辑。

```csharp
using UnityEngine;

public class BattleManager : MonoBehaviour
{
    void Start()
    {
        Debug.Log("=== 战斗开始 ===");

        // 1. 实例化不同的敌人
        EnemyAI monsterA = new MeleeEnemy();
        EnemyAI monsterB = new RangedEnemy();

        // 2. 执行它们的行为流程
        Debug.Log(">>> 轮到近战兵行动：");
        monsterA.ExecuteTurn();

        Debug.Log(">>> 轮到远程兵行动：");
```

## 5. 核心原理与优缺点总结

### 💡 它解决了什么痛点？
解决了 **“逻辑重复”** 和 **“流程混乱”** 。它通过把相同的代码提取到父类，不仅减少了重复工作，还通过“框架强制约束”确保了子类不会漏掉关键步骤。

### ✅ 优点
1.  **代码复用：** 相同的部分只写一遍。
2.  **控制权反转：** 由父类控制整体流程（控制大局），子类只负责具体实现（细节）。
3.  **便于维护：** 修改大框架只需要动基类一个地方。

### ❌ 缺点
1.  **框架限制：** 如果某个特殊的敌人不需要“移动”这一步，它也得强制实现这个方法（或者写个空实现）。
2.  **继承局限：** C# 只能单继承，如果用了模板方法占据了基类位置，就没法继承别的类了。

### 🎮 Unity 中的其他常见场景
* **UI 面板生命周期：** 定义 `Open()` 模板方法，流程为：播放入场动画 -> 初始化数据 -> 开启交互。子类只需实现各自的数据初始化即可。
* **资源加载流程：** 检查路径 -> 加载文件 -> **解析格式（变化的）** -> 回调通知。
* **关卡加载：** 显示加载条 -> 清理旧场景 -> **加载新物体（变化的）** -> 关闭加载条。

### ⚠️ 导师点拨：它和“策略模式”的区别
一句话点明核心区别：
> **模板方法模式**是基于**继承**的（在父类里定死步骤，子类填空）；而**策略模式**是基于**组合**的（把整个算法做成一个插件，随时可以拔掉换个新的）。

如果你发现你的代码里有很多“大同小异”的流程，那么模板方法模式就是你的最佳选择！加油！🚀
