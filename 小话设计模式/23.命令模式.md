你好！我是你的 Unity 开发导师。今天我们要攻克的是游戏开发中非常强大的**命令模式（Command Pattern）**。

如果说状态模式是让角色“变脸”，那么命令模式就是把角色的每一个动作都封装成一个 **“包裹”** 。有了这些包裹，你就拥有了操控时间的魔力。

---

## 1. 通俗易懂的例子：餐厅点餐 📝

想象你在一家繁忙的餐厅吃饭。

* **❌ 错误的做法（不使用模式）：**
    你直接冲进厨房，对着厨师大喊：“我要一份蛋炒饭！”
    * **麻烦：** 厨师正在炒菜，被你一喊手抖了。如果你突然不想要了，或者想换成面条，你得跟厨师吵架。如果有 10 个客人都冲进来喊，厨房就炸锅了。这里你（请求者）和厨师（执行者）**死死地耦合**在一起。

* **✅ 正确的做法（命令模式）：**
    你坐在位子上，在**点菜单（Command 对象）** 上写下“蛋炒饭”，交给服务员。
    * **巧妙之处：** 服务员把这张单子插在后厨的架子上。
    * **结果：** 厨师可以按照单子的顺序一张张做（队列）。如果你想反悔，只要单子还没下锅，服务员可以把单子撤回来（撤销 Undo）。老板甚至可以把今天的单子全留着，复盘今天卖了多少份（回放 Replay）。

## 2. Unity 游戏实战场景：策略或解谜游戏的“回溯”与“指令集” 🔄

在 Unity 中，**需要“撤销”功能的游戏**（如《时空幻境 Braid》或策略游戏的动作序列）是命令模式的最佳舞台。

* **为什么要用？**
    假设你在做一个推箱子游戏。如果玩家推错了一步，想按 `Ctrl+Z` 回到上一步。
* **痛点：**
    如果你只是简单地记录坐标，那以后如果有“推、拉、跳、炸”多种行为，你的撤销逻辑会写得异常复杂。
* **应用效果：**
    我们将“向上移动”封装成一个命令。这个命令不仅知道怎么执行 `Execute()`，还知道怎么反向操作 `Undo()`。我们只需要一个 `Stack`（栈）来存放这些单子，撤销就变成了点击一下“撤回最后一张单子”。

## 3. C# 代码实现（分步讲解）

### 第一步：定义命令接口（所有“单子”的标准）

```csharp
// 📜 命令接口：规定了所有的命令必须能“干活”和“反悔”
public interface ICommand
{
    void Execute(); // 执行动作
    void Undo();    // 撤销动作
}
```

### 第二步：编写具体的实现类（比如：移动命令）

```csharp
using UnityEngine;

// 🏃 移动命令：把“往哪儿走”封装成包裹
public class MoveCommand : ICommand
{
    private Transform _transform; // 谁动？
    private Vector3 _direction;   // 往哪动？
    private float _distance;      // 动多远？

    public MoveCommand(Transform target, Vector3 dir, float dist)
    {
        _transform = target;
        _direction = dir;
        _distance = dist;
    }

    // 执行：把物体推过去
    public void Execute()
    {
        _transform.position += _direction * _distance;
        Debug.Log($"[执行] {_transform.name} 向 {_direction} 移动了 {_distance} 米");
    }

    // 撤销：往反方向挪回来
    public void Undo()
    {
        _transform.position -= _direction * _distance;
        Debug.Log($"[撤销] {_transform.name} 已回到原位 ↩️");
    }
}
```

### 第三步：编写命令管理器（服务员/记录员）

```csharp
using System.Collections.Generic;

// 🗄️ 命令管理器：负责存放和处理所有的命令单子
public class CommandInvoker
{
    // 用一个栈来记录历史，方便“后进先出”的撤销
    private Stack<ICommand> _history = new Stack<ICommand>();

    public void DoCommand(ICommand cmd)
    {
        cmd.Execute();
        _history.Push(cmd); // 记在账本上
    }

    public void UndoCommand()
    {
        if (_history.Count > 0)
        {
            ICommand lastCmd = _history.Pop();
            lastCmd.Undo();
        }
    }
}
```

## 4. 在 Unity 中如何使用？

```csharp
using UnityEngine;

public class CommandTest : MonoBehaviour
{
    private CommandInvoker _invoker = new CommandInvoker();

    void Update()
    {
        // 1. 点击 W 键，发出一张“向上移动”的单子
        if (Input.GetKeyDown(KeyCode.W))
        {
            ICommand moveUp = new MoveCommand(transform, Vector3.up, 1f);
            _invoker.DoCommand(moveUp);
        }

        // 2. 点击 Z 键，进行撤销（后悔药）
        if (Input.GetKeyDown(KeyCode.Z))
        {
            _invoker.UndoCommand();
        }
    }
}

/* 预期 Debug.Log 输出结果：
1. (按下 W) [执行] Cube 向 (0, 1, 0) 移动了 1 米
2. (按下 W) [执行] Cube 向 (0, 1, 0) 移动了 1 米
3. (按下 Z) [撤销] Cube 已回到原位 ↩️
*/
```

## 5. 核心原理与优缺点总结

### 💡 它解决了什么痛点？
解决了 **“请求发送者”与“请求接收者”之间的耦合** 。它把一个请求变成了一个可以在程序里到处传递、存入列表、甚至是延迟执行的**独立对象**。

### ✅ 优点
1.  **完美支持撤销/重做：** 历史记录一清二楚。
2.  **延迟执行：** 可以先把命令攒着，到下一帧或者某个时间点一起执行。
3.  **代码复用：** 比如“攻击”命令可以给玩家用，也可以给 AI 用。

### ❌ 缺点
1.  **类变多了：** 每个小动作都要写一个类。
2.  **管理开销：** 如果游戏中每帧产生上万个命令，需要注意内存分配和回收。

### 🎮 Unity 中的其他常见场景
* **按键映射：** 玩家可以自定义按键。比如把“空格”绑定到 `JumpCommand` 还是 `ShootCommand`。
* **网络游戏预预测：** 客户端先执行命令，如果服务器返回结果不符，再调用 `Undo` 纠正。
* **过场动画脚本：** 按顺序执行一连串的“摄像机移动”、“NPC 说话”、“播放特效”命令。

### ⚠️ 导师点拨：它和“策略模式”的区别
一句话点明核心区别：
> **策略模式** 是关注 **“怎么做”**（比如：用什么算法寻路）；而 **命令模式** 是关注 **“做什么”**（比如：把这次移动当作一个任务存起来）。
