你好！我是你的 Unity 开发导师。今天我们要聊的是设计模式里的“外交官”——**中介者模式（Mediator Pattern）**。

在 Unity 开发中，随着项目变大，各个系统（UI、玩家、敌人、音效）之间往往会互相“加微信”，导致代码最后变成一团乱麻。中介者模式就是为了切断这些复杂的私聊，建立一个**中央调度室**。

---

## 1. 通俗易懂的例子：机场塔台 ✈️

想象一个繁忙的机场，有几十架飞机准备起飞或降落。

* **❌ 错误的做法（不使用模式）：**
    每架飞机的飞行员都得在无线电里跟**其他所有**飞行员沟通：“老兄，我要降落了，你让一下”、“我也要降落，你先等等”。
    * **麻烦：** 如果有 50 架飞机，每架飞机都要维护其他 49 架飞机的联系方式。只要新来一架飞机，所有人都要更新通讯录。这叫 **“网状耦合”**，稍有不慎就会撞机。
* **✅ 正确的做法（中介者模式）：**
    机场建立一个**塔台（中介者）**。
    * **巧妙之处：** 飞行员只需要跟塔台说话：“塔台，请求降落”。塔台观察全局后告诉他：“可以降落”或者“在云层待命”。飞行员之间不需要互相认识。
    * **结果：** 飞机只跟塔台耦合。塔台掌控全局，协调所有复杂的逻辑。

## 2. Unity 游戏实战场景：复杂的 UI 面板联动 🖥️

在 Unity 中，**复杂的 UI 系统**是中介者模式最典型的舞台。

* **场景描述：**
    假设你有一个“角色装备界面”，里面有：
    1.  **包裹列表**（点击物品）。
    2.  **属性面板**（显示数值变化）。
    3.  **装备按钮**（点击后穿上）。
    4.  **音效系统**（播放穿戴声）。
* **为什么适合？**
    如果你让“包裹列表”直接去调用“属性面板”更新文字，再去调用“音效系统”播声音，代码会非常难维护。使用中介者（比如一个 `UIManager`），所有组件只向中介者汇报“我被点了”，由中介者统一分发指令。

## 3. C# 代码实现（分步讲解）

### 第一步：定义中介者接口与组件基类

```csharp
using UnityEngine;

// 📜 中介者接口：定义了组件如何向中介者汇报
public interface IMediator
{
    void OnNotify(ComponentBase sender, string evnt);
}

// 🧱 组件基类：所有参与联动的模块都要持有中介者的引用
public abstract class ComponentBase
{
    protected IMediator _mediator;

    public ComponentBase(IMediator mediator)
    {
        _mediator = mediator;
    }
}
```

### 第二步：编写具体的组件（飞机/UI元素）

```csharp
// 🎒 背包列表组件
public class Inventory : ComponentBase
{
    public Inventory(IMediator mediator) : base(mediator) { }

    public void SelectItem()
    {
        Debug.Log("[背包] 玩家选中了一个神话头盔。");
        // 📢 向中介者汇报事件
        _mediator.OnNotify(this, "ItemSelected");
    }
}

// ⚔️ 装备按钮组件
public class EquipButton : ComponentBase
{
    public EquipButton(IMediator mediator) : base(mediator) { }

    public void SetVisible(bool visible)
    {
        string status = visible ? "显示" : "隐藏";
        Debug.Log($"[装备按钮] 按钮现在已{status}。");
    }
}
```

### 第三步：实现具体的中心调度员（塔台）

```csharp
// 🏢 UI中介者：负责协调背包和按钮
public class UIMediator : IMediator
{
    private Inventory _inventory;
    private EquipButton _equipButton;

    // 初始化时把需要管理的组件凑齐
    public void Initialize(Inventory inv, EquipButton btn)
    {
        _inventory = inv;
        _equipButton = btn;
    }

    // 🧠 核心逻辑：在这里处理复杂的联动逻辑
    public void OnNotify(ComponentBase sender, string evnt)
    {
        if (evnt == "ItemSelected")
        {
            Debug.Log("[中介者] 收到物品选中消息，正在协调 UI...");
            _equipButton.SetVisible(true); // 让装备按钮亮起来
        }
    }
}
```

## 4. 在 Unity 中如何使用？

我们将中介者模式挂载到一个控制器上进行测试。

```csharp
public class MediatorTest : MonoBehaviour
{
    void Start()
    {
        // 1. 实例化中介者
        UIMediator uiTower = new UIMediator();

        // 2. 实例化组件，并把中介者传给它们
        Inventory myInventory = new Inventory(uiTower);
        EquipButton myBtn = new EquipButton(uiTower);

        // 3. 让中介者认识这些组件
        uiTower.Initialize(myInventory, myBtn);

        Debug.Log("--- 模拟玩家操作 ---");

        // 4. 模拟玩家点击背包物品
        myInventory.SelectItem();
    }
}

/* 预期 Debug.Log 输出结果：
1. --- 模拟玩家操作 ---
2. [背包] 玩家选中了一个神话头盔。
3. [中介者] 收到物品选中消息，正在协调 UI...
4. [装备按钮] 按钮现在已显示。
*/
```

## 5. 核心原理与优缺点总结

### 💡 它解决了什么痛点？
解决了对象之间 **“多对多”** 的交叉引用。它将对象之间的交互行为封装在单一的中介者对象中，使原本杂乱无章的依赖关系变成了清晰的 **“星形结构”**。

### ✅ 优点
* **极致解耦：** 各个组件互相不认识，只认识中介者。
* **易于维护：** 修改逻辑只需要改中介者一个类，而不需要翻遍整个项目。
* **符合迪米特法则：** 一个对象应当对其他对象有尽可能少的了解。

### ❌ 缺点
* **中介者臃肿：** 如果系统太复杂，中介者会变成一个无所不知、长达几千行的“上帝类”。

### 🎮 Unity 中的其他常见场景
* **关卡管理器（LevelManager）：** 负责协调玩家死亡、怪物刷新、倒计时 UI 和胜利特效。
* **战斗结算系统：** 协调金币增加、经验条上涨、成就弹出和存档触发。

### ⚠️ 导师点拨：它和“观察者模式”的区别
一句话点明核心区别：
> **观察者模式**是“发广播”，消息是**单向**散播的，发完就走；而**中介者模式**是“双向调度”，它不仅接收消息，还要根据业务逻辑去**主动操控**其他对象。
