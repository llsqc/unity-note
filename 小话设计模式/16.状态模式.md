## 1. 通俗易懂的例子：水的三态变化 💧

想象你手里有一杯物质，它的名字叫“水”。

* **❌ 错误的做法（不使用模式）：**
    你写了一个 `Water` 类，里面有一个 `Touch()` 方法。为了判断摸上去是什么感觉，你得写一堆判断：
    * “如果当前温度低于 0 度，它是冰，摸上去很硬。”
    * “如果当前温度在 0 到 100 度，它是水，摸上去很凉。”
    * “如果当前温度高于 100 度，它是蒸汽，摸上去很烫。”
    * **麻烦：** 如果你还要增加一个“超临界状态”或者“等离子态”，你得把这个 `Touch()` 方法翻出来大改特改，逻辑越堆越多，最后连你自己都看不懂了。

* **✅ 正确的做法（状态模式）：**
    你直接定义三个独立的**“状态包”**：【冰块包】、【液体包】、【蒸汽包】。
    * 当你把“水”放进冰箱，它就把手里的逻辑切换到了【冰块包】。
    * **巧妙之处：** 此时你再调用 `Touch()`，“水”本身不思考，它直接把任务甩给当前的“包”。【冰块包】会直接告诉你：“硬邦邦！”。
    * **结果：** “水”类变得非常干净，它只负责切换包，而每个包只负责写好自己的逻辑。

## 2. Unity 游戏实战场景：主角动作状态机 🏃‍♂️

在 Unity 中，**主角的动作控制**是状态模式最对口的场景。

* **为什么要用？**
    主角在不同状态下的行为逻辑完全不同：
    * **待机状态（Idle）：** 播放呼吸动画，监听方向键，按空格切换到“跳跃”。
    * **奔跑状态（Run）：** 播放跑步动画，改变物理位移，松开按键切换到“待机”。
    * **跳跃状态（Jump）：** 播放升空动画，不监听左右移动（假设），落地后切换回“待机”。
* **应用效果：**
    使用状态模式后，你不需要在 `Update` 里写：`if(isJumping) {...} else if(isRunning) {...}`。你只需要告诉状态机：`currentState.Execute()`。逻辑清晰得像在看说明书。

## 3. C# 代码实现（分步讲解）

### 第一步：定义状态接口（所有状态必须遵守的契约）

```csharp
// 📜 状态接口：规定了每个状态都要有的行为
public interface IPlayerState
{
    void Enter();  // 进入状态时执行一次
    void Update(); // 在状态中每帧执行
    void Exit();   // 离开状态前执行一次
}
```

### 第二步：编写具体的实现类（各种动作包）

```csharp
using UnityEngine;

// 🧘 待机状态
public class IdleState : IPlayerState
{
    private PlayerController _player;
    public IdleState(PlayerController player) => _player = player;

    public void Enter() => Debug.Log("--- 进入待机状态：开始呼吸动画 ---");

    public void Update()
    {
        // 键盘输入检测：如果按了 WASD，就告诉玩家切换到跑步状态
        if (Input.GetAxis("Horizontal") != 0 || Input.GetAxis("Vertical") != 0)
        {
            _player.SwitchState(_player.RunState);
        }
    }

    public void Exit() => Debug.Log("--- 离开待机状态 ---");
}

// 🏃 奔跑状态
public class RunState : IPlayerState
{
    private PlayerController _player;
    public RunState(PlayerController player) => _player = player;

    public void Enter() => Debug.Log("--- 进入奔跑状态：开始跑步烟尘特效 ---");

    public void Update()
    {
        // 模拟位移逻辑
        Debug.Log("正在奔跑中... 💨");

        // 如果没有按键输入，切回待机
        if (Input.GetAxis("Horizontal") == 0 && Input.GetAxis("Vertical") == 0)
        {
            _player.SwitchState(_player.IdleState);
        }
    }

    public void Exit() => Debug.Log("--- 离开奔跑状态：停止特效 ---");
}
```

## 4. 在 Unity 中如何使用？

我们通过 `PlayerController`（上下文类）来驱动状态的切换。

```csharp
using UnityEngine;

public class PlayerController : MonoBehaviour
{
    // 持有当前正在执行的状态
    private IPlayerState _currentState;

    // 提前实例化好状态对象，方便切换
    public IdleState IdleState;
    public RunState RunState;

    void Start()
    {
        // 1. 初始化状态
        IdleState = new IdleState(this);
        RunState = new RunState(this);

        // 2. 设置初始状态
        _currentState = IdleState;
        _currentState.Enter();
    }

    void Update()
    {
        // 3. 核心精髓：每帧只需要执行当前状态的逻辑即可！
        // 里面写了什么？PlayerController 根本不关心
        _currentState.Update();
    }

    // 4. 提供切换状态的方法
    public void SwitchState(IPlayerState newState)
    {
        _currentState.Exit();  // 旧状态退场
        _currentState = newState;
        _currentState.Enter(); // 新状态进场
    }
}

/* 预期 Debug.Log 输出结果：
1. --- 进入待机状态：开始呼吸动画 ---
2. (当按下方向键时...)
3. --- 离开待机状态 ---
4. --- 进入奔跑状态：开始跑步烟尘特效 ---
5. 正在奔跑中... 💨
*/
```

## 5. 核心原理与优缺点总结

### 💡 它解决了什么痛点？
解决了 **“分支逻辑爆炸”** 的问题。它把“由于状态改变而导致的行为改变”封装到了独立的对象中，让复杂的逻辑变得可拆卸、可维护。

### ✅ 优点
1.  **结构清晰：** 每一个状态都是一个独立的类，找 Bug 只需要去对应的类里看。
2.  **扩展性极强：** 增加一个“潜行状态”？新建个类实现接口，然后在 `PlayerController` 里加个引用就行，完全不需要改动原来的 `Idle` 或 `Run` 逻辑。
3.  **遵循 OCP：** 对扩展开放，对修改关闭。

### ❌ 缺点
1.  **类文件增多：** 如果状态非常多，你的项目里会多出很多小脚本。
2.  **逻辑分散：** 某些简单的状态切换如果也写成类，可能会觉得有点“杀鸡用牛刀”。

### 🎮 Unity 中的其他常见场景
* **敌人 AI：** 巡逻（Patrol）、追踪（Chase）、攻击（Attack）之间的切换。
* **游戏流程控制：** 菜单界面、加载中、游戏进行中、结算画面。
* **武器系统：** 填弹状态、开火状态、冷却状态。

### ⚠️ 导师点拨：它和“策略模式”的区别
一句话点明核心区别：
> **策略模式**通常由**外部**（客户端）决定使用哪种算法；而**状态模式**通常是**对象内部**自己感知条件并**主动切换**（比如 `IdleState` 内部检测按键并决定切到 `RunState`）。
