你好！我是你的 Unity 开发导师。今天我们要聊的是面向对象设计原则中最基础、也最重要的一条规则：**单一职责原则（Single Responsibility Principle, 简称 SRP）**。

很多新手在写代码时，最容易犯的错误就是把所有的逻辑全塞进一个脚本里。今天我们就彻底把这个坏习惯改掉。严格按照你的要求，我们分 5 个模块来拆解它！

### 1. 通俗易懂的例子（生活隐喻）

**👨‍🍳 生活隐喻：街边小吃摊 vs 高级米其林餐厅**

想象你开了一家餐饮店。

*   ❌ **错误的做法（不使用 SRP）：**
    你招了一个“全能型”员工叫小明。小明同时负责：在门口迎客、在后厨炒菜、给客人端盘子、最后还要收银。
    **带来的麻烦：**
    有一天，收银系统升级了，小明为了学习新收银机，结果导致后厨的菜全烧糊了！又或者小明炒菜时切到了手，你的店连迎客和收银的人都没了，整个店直接瘫痪。这就是代码里的 **“上帝类（God Class）”**，牵一发而动全身。
*   ✅ **正确的做法（使用 SRP）：**
    你把工作拆分：招一个“迎宾员”、一个“厨师”、一个“服务员”、一个“收银员”。
    **巧妙之处：**
    收银系统升级，只需要培训“收银员”，厨师照样炒菜，互不干扰。厨师请假了，你只需要再招一个新厨师，收银和服务员完全不受影响。

**💡 核心思想：** 一个类（或脚本），**应该只有一个引起它变化的原因**。也就是“一人只干一件事”。

### 2. Unity 游戏实战场景

**🎮 典型实战场景：玩家角色控制（Player Controller）**

**为什么适合？**
在 Unity 中，新手最喜欢建一个叫 `Player.cs` 的脚本，挂在主角身上。然后在这个脚本里写：按键输入、移动逻辑、扣血逻辑、播放动画、播放音效、甚至是UI更新。

当游戏做到中后期，这个脚本可能会有 3000 行！
这时你想给游戏增加一个“手柄操作”，你需要在这个 3000 行的文件里找输入逻辑去改，稍不注意就把角色的“扣血逻辑”给改出 Bug 了。而且，如果是团队开发，美术在调动画代码，你在调移动代码，你们提交时必定发生严重的**代码冲突**。

**遵循单一职责的做法：**
把 `Player.cs` 拆分成：
1. `PlayerInput`（只负责读取键盘/手柄输入）
2. `PlayerMovement`（只负责处理位置和物理移动）
3. `PlayerHealth`（只负责血量计算和死亡）

### 3. C# 代码实现（分步讲解）

为了满足规范，我们将“输入”和“移动”拆开，用接口来解耦。

#### 第一步：定义相关的接口（明确职责边界）
我们先定义一个专门提供输入数据的接口。它的唯一职责就是“告诉别人现在该往哪走”。

```csharp
using UnityEngine;

// 🎯 定义输入接口：只负责传递输入数据，不管你怎么移动
public interface IInputProvider
{
    // 获取水平移动的输入值 (-1 到 1)
    float GetHorizontalInput();
    
    // 是否按下了跳跃键
    bool GetJumpInput();
}
```

#### 第二步：编写具体的实现类（输入类）
接下来，写一个专门负责读取玩家键盘操作的类。如果以后换成手机摇杆，只需新建个类，不用改现有的代码。

```csharp
// ⌨️ 键盘输入类：唯一的职责就是监听玩家的键盘操作
public class KeyboardInput : MonoBehaviour, IInputProvider
{
    public float GetHorizontalInput()
    {
        // 监听 A/D 键或左右方向键
        return Input.GetAxisRaw("Horizontal");
    }

    public bool GetJumpInput()
    {
        // 监听空格键
        return Input.GetKeyDown(KeyCode.Space);
    }
}
```

#### 第三步：编写具体的实现类（移动类）
这个类只负责控制 Transform 或 Rigidbody，它**不关心**指令是来自键盘、手柄还是 AI，它只管“移动”。

```csharp
// 🏃‍♂️ 角色移动类：唯一的职责就是控制游戏物体的移动
public class PlayerMovement : MonoBehaviour
{
    public float moveSpeed = 5f;
    
    // 依赖于接口，而不是具体的 KeyboardInput，进一步解耦！
    private IInputProvider _inputProvider;

    private void Awake()
    {
        // 自动获取挂在同一个 GameObject 上的输入组件
        _inputProvider = GetComponent<IInputProvider>();
    }

    // 移动逻辑
    public void Move()
    {
        if (_inputProvider == null) return;

        // 1. 获取输入（它不知道是谁传来的，只管用）
        float direction = _inputProvider.GetHorizontalInput();
        
        // 2. 执行移动
        transform.Translate(Vector3.right * direction * moveSpeed * Time.deltaTime);

        // 3. 执行跳跃
        if (_inputProvider.GetJumpInput())
        {
            Jump();
        }
    }

    private void Jump()
    {
        // 这里只是做演示，实际可能会给 Rigidbody 施加力
        Debug.Log("🚀 执行了跳跃逻辑！");
    }
}
```

### 4. 在 Unity 中如何使用？

我们创建一个测试脚本，挂载到一个新建的 Cube（模拟玩家）上。
注意：你需要在这个 Cube 上同时挂载 `KeyboardInput` 和 `PlayerMovement`。

```csharp
using UnityEngine;

// 🎮 客户端测试类：负责把各个模块组合起来运行
public class GameTest : MonoBehaviour
{
    // 引用移动组件
    private PlayerMovement _playerMovement;

    void Start()
    {
        // 确保物体上挂载了所需组件
        if (GetComponent<KeyboardInput>() == null)
        {
            gameObject.AddComponent<KeyboardInput>();
            Debug.Log("🔧 动态添加了职责1：输入组件");
        }

        if (GetComponent<PlayerMovement>() == null)
        {
            _playerMovement = gameObject.AddComponent<PlayerMovement>();
            Debug.Log("🔧 动态添加了职责2：移动组件");
        }
        else
        {
            _playerMovement = GetComponent<PlayerMovement>();
        }
        
        /* 预期输出：
         * 🔧 动态添加了职责1：输入组件
         * 🔧 动态添加了职责2：移动组件
         */
    }

    void Update()
    {
        // 在每一帧调用移动类的 Move 方法
        // 移动类内部会自动去向输入类要数据，各司其职
        if (_playerMovement != null)
        {
            _playerMovement.Move();
        }
        
        /* 运行流程预期：
         * 1. 游戏运行，Cube 停在原地不动。
         * 2. 当你按下右方向键(D键)，Cube 向右平移。
         * 3. 当你按下空格键，控制台会输出：🚀 执行了跳跃逻辑！
         */
    }
}
```

### 5. 核心原理与优缺点总结

**💡 解决了什么痛点？**
解决了 **“代码一团糟，改一个Bug引发十个新Bug”** 的痛点。把庞大的“上帝类”拆分成小巧、精悍、单一的积木块。

**👍 优点：**
*   **高可读性：** 想改什么功能，直接定位到具体名字的脚本文件，不用在几千行代码里滑鼠标。
*   **易于测试和复用：** 比如 `PlayerMovement` 写的很好，你想做个敌人 AI，直接把 `PlayerMovement` 挂给敌人，然后写个 `AIInput` 接口传给它就行了，代码一行不用改！
*   **团队友好：** 你写移动逻辑，同事写输入逻辑，两人修改的是不同的脚本文件，提交代码时绝对不会发生 Git 冲突。

**👎 缺点：**
*   **类的数量激增：** 原本 1 个脚本搞定的事，现在变成了 3-4 个脚本，微小项目可能会觉得有点过度设计。
*   **组件通信成本：** 拆分后，组件之间需要通过 `GetComponent` 或事件委托来相互调用，增加了一点点拼装的麻烦。

**🎯 Unity 中的其他常见应用场景：**
1. **音频系统：** 不要让怪物自己去写 `AudioSource.Play()`。让怪物抛出死亡事件，由一个专门的 `AudioManager` 负责播放声音。
2. **存档系统：** 玩家类只负责游玩逻辑，存读档交给专门的 `SaveManager` 去负责写入 JSON/本地文件。
3. **UI 数据分离（MVC架构）：** UI 面板只负责显示文字和血条，不负责计算升级所需经验，计算逻辑交给具体的 Model 数据类。

**⚠️ 容易混淆的设计模式：**
单一职责原则（SRP）非常容易和 **接口隔离原则（ISP）** 混淆。
*   **一句话点明区别：** **单一职责**是强调“一个类只**做**一件事”（关注内部实现）；而**接口隔离**强调的是“不强迫别人**认领**他们不需要的功能”（关注对外暴露）。
