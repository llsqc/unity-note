**RPC（Remote Procedure Call，远程过程调用）** 是一种通信机制，让你可以**像调用本地函数一样，去调用另一台机器上的函数**。

---

## 一句话理解

👉 RPC = “远程调用函数，但写起来像本地调用”

---

## 举个简单例子 🌐

假设你在做一个联网游戏（比如用 Unity）：

```csharp
TakeDamage(10);
```

如果是本地函数 👉 就是自己调用自己  
如果是 RPC 👉 实际发生的是：

👉 客户端调用 →  
👉 通过网络发送请求 →  
👉 服务器执行 `TakeDamage(10)` →  
👉 返回结果

---

## RPC 在做什么？

它帮你做了这些复杂事情：

- 📡 网络通信（发送/接收数据）
    
- 📦 数据序列化（把参数变成可传输格式）
    
- 🔄 调用远程函数
    
- 📬 返回结果
    

👉 你不用手写 Socket 细节

---

## 常见使用场景

### 1️⃣ 游戏开发（非常常见）

- 客户端 → 服务器：发技能、移动、攻击
    
- 服务器 → 客户端：同步状态
    

例如：

- 玩家开枪 → RPC 通知服务器
    
- 服务器计算命中 → RPC 通知所有玩家
    

---

### 2️⃣ 微服务 / 后端开发

比如：

- 用户服务 调用 支付服务
    
- 订单系统 调用 库存系统
    

常见技术：

- gRPC
    
- Thrift
    
- Dubbo
    

---

### 3️⃣ 分布式系统

- 不同机器之间协同工作
    
- 像调用本地函数一样调用远程逻辑
    

---

## RPC vs HTTP API（很多人会混）

|对比|RPC|HTTP API（REST）|
|---|---|---|
|调用方式|像函数|像访问 URL|
|语义|强（方法名）|弱（资源 + 动词）|
|常见协议|TCP / HTTP2|HTTP|
|使用感受|更像写代码|更像请求接口|

👉 简单说：

- RPC 更“程序员内部调用”
    
- REST 更“对外开放接口”
    

---

## 在 Unity 里的 RPC（补充）

用网络框架（如 Mirror / Netcode）时：

```csharp
[ClientRpc]
void RpcTakeDamage(int damage)
{
    // 在所有客户端执行
}
```

👉 调用后会在所有客户端执行这个函数

RPC 的核心就是：

✅ 屏蔽网络细节  
✅ 像调用本地函数一样调用远程逻辑  
✅ 广泛用于游戏、后端、分布式系统

# 🧠 RPC 的核心原理（一句话）

> **把函数调用 → 变成网络消息 → 在远程还原并执行**

---

# 🔧 拆成 4 个步骤（关键）

假设在 Unity 里调用一个 RPC：

```csharp
RpcTakeDamage(10);
```

---

## 1️⃣ 拦截函数调用（最关键的一步）

这个函数其实不会“直接执行”。

因为你写了类似：

```csharp
[ClientRpc]
void RpcTakeDamage(int damage)
```

👉 框架会做一件事：

**拦截这个函数调用**

把它变成：

> ❌ 普通函数调用  
> ✅ “我要发一条网络消息”

---

## 2️⃣ 序列化参数（打包数据）

```id="c8p3qh"
函数名：RpcTakeDamage
参数：10
```

👉 会被打包成：

```id="z1n9ka"
{
  "method": "RpcTakeDamage",
  "params": [10]
}
```

👉 这个过程叫：  
**序列化（Serialization）**

---

## 3️⃣ 通过网络发送

👉 用底层协议发送（例如）：

- TCP
    
- UDP
    

发送给：

- 服务器
    
- 或其他客户端（通常经过服务器）
    

---

## 4️⃣ 远程还原并执行

另一端收到数据后：

```id="y2k7fw"
{
  "method": "RpcTakeDamage",
  "params": [10]
}
```

👉 做三件事：

1. 找到函数 `"RpcTakeDamage"`
    
2. 解析参数 `10`
    
3. 执行函数
    

```csharp
RpcTakeDamage(10);
```

👉 **这时候才真正执行！**

---

# 🔁 用一句流程串起来

```id="5j3f2p"
调用函数
→ 被框架拦截
→ 打包成数据
→ 网络发送
→ 远程解析
→ 执行函数
```

---

# ⚙️ 底层关键技术点

## ✅ 1. 序列化 / 反序列化

把数据变成可传输格式（JSON / 二进制）

---

## ✅ 2. 网络通信

通过 Socket / UDP / TCP 传输数据

---

## ✅ 3. 函数映射表（很关键）

远程怎么知道调用哪个函数？

👉 内部会有一个类似：

```id="mj8r5s"
ID 1 → RpcTakeDamage
ID 2 → RpcPlayEffect
```

👉 实际传输可能是：

```id="7x4cpa"
{ id: 1, params: [10] }
```

👉 更高效（不传字符串）

---

## ✅ 4. 代码生成 / 反射（实现方式）

不同框架实现方式不同：

- 🧩 反射（Reflection）
    
- ⚡ IL 代码生成（更快）
    
- 🏗 编译期生成代码（Unity Netcode 常用）
    

---

# 🎮 为什么 RPC 看起来像“函数”？

👉 因为框架帮你做了：

- 拦截调用
    
- 自动打包
    
- 自动发送
    
- 自动执行
    

👉 **你看到的是函数，本质是消息系统**

---

# ⚠️ 本质提醒（非常重要）

👉 RPC ≠ 函数调用  
👉 RPC = **消息 + 远程执行**

所以它有这些特性：

- ⏱ 有延迟（网络）
    
- ❌ 可能丢失（UDP）
    
- 🔁 可能乱序
    
- 🔒 需要权限控制
    

---

# ✅ 最终总结（工程理解）

👉 RPC 的本质就是：

> **用“网络消息机制”模拟“函数调用体验”**
